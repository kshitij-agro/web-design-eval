# Design-to-Code Eval Recipe — Plan & Findings

Living document. Captures the work-trial brief, everything learned about the
Harbor framework, the local environment, and the proposed recipe design.
Update as decisions land.

---

## 1. The brief

Build a scalable RL-environment recipe that grades a coding agent's ability to
turn a website screenshot into HTML+CSS. Each generated task:

- Shows the agent a screenshot of a single-page website.
- Asks it to replicate the design in HTML+CSS (functionality ignored).
- Grades the result on a **continuous spectrum** (not pass/fail).

Constraints:

- Implemented as a Harbor task / dataset.
- Tested with Claude Code + **Opus 4.6**.
- **No crawling** of existing websites — designs must be synthesised.
- A good distribution across website *types*.

Deliverable:

- The recipe (automation pipeline).
- 1–2 final tasks generated by the recipe.
- A visual report: 10× Opus 4.6 runs per task, showing how well the grader
  scores the results.

## 2. Harbor framework — the task contract

Confirmed from the [docs](https://harborframework.com/docs) and the
[harbor-cookbook](https://github.com/harbor-framework/harbor-cookbook) recipes
(`simple-task`, `multi-reward`).

### 2.1 Task layout

```
<task-name>/
├── task.toml                 # config + metadata
├── instruction.md            # the prompt the agent receives
├── environment/
│   └── Dockerfile            # (or docker-compose.yaml)
│   └── <files baked in>      # COPY'd into the image
├── solution/                 # optional reference implementation
│   └── solve.sh
└── tests/
    └── test.sh               # entrypoint; must write reward
    └── <python tests, refs>
```

### 2.2 Container paths (linux tasks)

| Path | Purpose |
|------|---------|
| `/app` | Agent CWD (set by `WORKDIR`). |
| `/tests/` | Test scripts copied at verify time. |
| `/solution/` | Oracle solution copied at verify time. |
| `/logs/agent/` | Agent-generated logs. |
| `/logs/verifier/` | Where the verifier writes rewards. |

### 2.3 Reward contract — **continuous is supported**

- `/logs/verifier/reward.txt` → plain text, integer **or float**. The cookbook
  examples use 0/1, but float values are explicitly accepted.
- `/logs/verifier/reward.json` → multi-dimensional, e.g.
  `{"correctness": 0.8, "performance": 0.4}`.
- We will write **both**: a single weighted scalar in `reward.txt` and the
  per-component breakdown in `reward.json`.

### 2.4 Instruction delivery

The contents of `instruction.md` are passed to the agent as its starting
prompt. The agent operates inside the container with its normal tools (Read,
Edit, Write, Bash). For Claude Code specifically, the `Read` tool supports
image files and presents them visually to Opus 4.6. **This is how the
screenshot reaches the model — no special glue required.**

### 2.5 Useful CLI

- `harbor task init org/name` — scaffold a task.
- `harbor run -a claude-code -m claude-opus-4-6 -k N -p path/to/task` — run
  trials.
- `harbor task check` — quality lint.
- `harbor task debug` — inspect failing trials.
- `harbor view` — browse trial trajectories in a web UI.

### 2.6 Reference example: `multi-reward` `tests/test.sh`

```bash
#!/bin/bash
apt-get update && apt-get install -y curl
curl -LsSf https://astral.sh/uv/0.9.7/install.sh | sh
source $HOME/.local/bin/env

correctness=0; performance=0
uvx --with pytest==8.4.1 --with pytest-json-ctrf==0.3.5 \
    pytest --ctrf /logs/verifier/ctrf.json /tests/test_correctness.py -rA \
    && correctness=1
uvx --with pytest==8.4.1 --with pytest-json-ctrf==0.3.5 \
    pytest --ctrf /logs/verifier/ctrf_performance.json /tests/test_performance.py -rA \
    && performance=1

python3 -c "
import json
json.dump({'correctness': $correctness, 'performance': $performance},
          open('/logs/verifier/reward.json', 'w'))
"
```

We'll follow this shape but write a **continuous** scalar instead of a
0/1 flag.

## 5. Recipe design

### 5.0 Tooling split: renderer vs. grader

It is worth being explicit because the two are easy to conflate:

| Concern | Tool | Used at |
|---------|------|---------|
| **Rendering** HTML → PNG | Playwright (headless Chromium) | Generation time (reference) **and** verify time (agent output) |
| **Scoring** PNG vs. PNG (continuous) | scikit-image SSIM + OpenCV (layout) + scipy (colour EMD) | Verify time only |

Playwright is *only* a renderer in this pipeline. It does ship a built-in
visual-comparison API (`expect(page).to_have_screenshot()` /
[`pixelmatch`](https://github.com/mapbox/pixelmatch)), but that is designed
for visual-regression testing — pass/fail against a pixel-count threshold,
not a continuous similarity score. We deliberately don't use it as the
grader. The continuous score is computed by the grader described in §7.

### 5.1 High-level flow

There are three phases. Note carefully which artefacts are produced when —
nothing in the verify phase is created by the agent except
`/app/output/index.html`.

#### Phase 1 — Generation (offline, once per task)

```
archetype × style × seed
        │
        ▼
LLM (Claude Sonnet) ─────────── reference index.html
        │
        ▼
Playwright render (1280×900, full_page) ────── reference.png
```

Outputs of this phase are static files on the author's machine.

#### Phase 2 — Bundling (offline, stamps a Harbor task dir)

The same `reference.png` is placed in **two** locations inside the task
directory, with different roles:

| Path in task dir | Path inside container at runtime | Visible to | Purpose |
|------------------|----------------------------------|------------|---------|
| `environment/reference.png` | `/app/reference.png` | **Agent** | The screenshot the agent must replicate. Baked into the image via `COPY` in the Dockerfile. |
| `tests/reference.png` | `/tests/reference.png` | **Verifier only** | Private ground truth used by the grader. Copied alongside `test.sh` at verify time; agent has no access. |

Plus the reference HTML lands in `solution/index.html` (oracle only — see §"oracle solution" discussion).

#### Phase 3 — `harbor run` (per trial, inside the container)

```
container starts with /app/reference.png already baked in
        │
        ▼
Claude Code (Opus 4.6) reads /app/reference.png
        │
        ▼
Claude Code writes /app/output/index.html      ← the only file the agent produces
        │
        ▼  (verify time)
tests/test.sh runs:
  1. Playwright renders /app/output/index.html  → /tmp/agent.png
  2. grader compares /tmp/agent.png             ← agent-rendered, fresh per trial
                vs /tests/reference.png         ← bundled at generation time, static
     (using SSIM + colour EMD + layout IoU + height penalty — see §7)
  3. write scalar to /logs/verifier/reward.txt
     write breakdown to /logs/verifier/reward.json
```

So at verify time, the *only* PNG generated inside the container is the
agent's render (`/tmp/agent.png`). `/tests/reference.png` is read-only,
bundled with the task, identical for every trial.

### 5.2 Diversity axes

| Axis | Examples | Count |
|------|----------|-------|
| **Archetype** | landing, pricing, blog post, dashboard, portfolio, ecommerce product, login form, news article, contact form, app marketing | ~10 |
| **Style** | minimalist, brutalist, glassmorphism, retro, dark-mode SaaS, editorial | ~6 |
| **Seed** | random sampling of palette, font pairing, copy variants | unbounded |

This product space gives ~60 distinct archetype×style buckets with arbitrary
copy/colour variation per seed — easily scales to hundreds of tasks.

### 5.3 Synthesis: LLM-generated reference HTML

**Why LLM, not pure procedural templates:** procedural gets us deterministic
output and zero API cost, but building 10 archetype × 6 style × N detail
templates by hand is a lot of CSS work for marginal visual realism. LLM
generation reaches "feels like a real website" in one prompt. Cost is small
(<$0.05/page on Sonnet) and only paid once per task at generation time.

**Generator prompt constraints (hard requirements):**

- Single self-contained `index.html`. Inline `<style>`. No external CSS files.
- No `<script>`. Static markup only.
- No external assets. SVG inline; images via data URIs or solid blocks. Web
  fonts only via Google Fonts `<link>`.
- Designed for a 1280×900 viewport.
- Must be valid HTML (parse-checked post-gen).
- Must produce a non-trivial render (post-gen check: rendered PNG isn't blank
  or near-monochrome).

Generated pages that fail post-gen validation are rejected and re-rolled with
a different seed.

### 5.4 Reference render

Headless Chromium via Playwright at fixed viewport 1280×900, **`full_page=True`**
so scrolling pages render in their entirety. Output: `reference.png`.

### 5.5 Task bundling

For each accepted (archetype, style, seed):

```
tasks/<archetype>-<style>-<seed>/
├── task.toml                  # filled-in metadata, timeouts, resources
├── instruction.md             # explicit "Read /app/reference.png, write /app/output/index.html"
├── environment/
│   ├── Dockerfile             # FROM mcr.microsoft.com/playwright/python:...
│   └── reference.png          # COPY'd to /app/reference.png
├── solution/
│   ├── solve.sh               # cp /solution/index.html /app/output/index.html
│   └── index.html             # the generator's reference HTML — oracle upper bound
└── tests/
    ├── test.sh                # entry: render, score, write reward
    ├── grade.py               # the continuous grader
    ├── reference.png          # private copy — agent doesn't see this path
    └── reference.html         # for optional DOM-distance metric
```

`solution/` exists so we can sanity-check the grader: running with the
`oracle` agent should yield a reward at or near 1.0. If not, the grader is
miscalibrated.

## 6. How the screenshot reaches the agent

Spelled out because this was a clarifying question.

1. **Generator** — Playwright renders `reference.png` at 1280×900.
2. **Bundle** — file placed at `environment/reference.png`. The Dockerfile
   does `COPY reference.png /app/reference.png` so it's baked into the image.
3. **Instruction** — `instruction.md` explicitly tells the agent to open it:
   > Open `/app/reference.png` with the Read tool (it is an image of a
   > website). Replicate the design as a single `/app/output/index.html`
   > using only HTML and inline CSS. Match layout, typography, colours, and
   > spacing as closely as possible. Functionality is not required.
4. **Agent runtime** — Claude Code's `Read` tool detects the PNG, loads it as
   an image, and presents it to Opus 4.6 (multimodal). The agent then writes
   HTML/CSS via `Edit`/`Write`.

### 6.1 Constraints this imposes

- **Image size:** keep PNGs reasonable (~1 MB). 1280×900 lossless PNG is
  comfortably under.
- **One viewport per task:** reference and agent renders both happen at the
  same viewport so the grader is comparing apples to apples.
- **Full-page render:** `full_page=True` so tall pages aren't truncated.
  Agent's render also uses `full_page=True`. Heights differ → height penalty
  in grader (see §7).
- **Colour fidelity:** sRGB end-to-end. Same Chromium build at gen and verify
  time (achieved by using the official Playwright base image).

### 6.2 Why we're not embedding the image in the prompt

We could base64-embed in `instruction.md`, but:

- Harbor passes the file in `environment/` to the container natively.
- Claude Code's `Read` tool already handles images.
- Keeping it on disk preserves the option for the agent to re-Read it
  (e.g., to re-check details mid-task) without bloating the prompt.

## 7. Continuous grader

Lives in `tests/grade.py`, invoked from `tests/test.sh`. Runs inside the task
container after the agent finishes.

### 7.1 Pipeline

The grader is two stages: **render**, then **score**. Playwright handles only
the render step (cf. §5.0); all continuous scoring is done in-process with
scikit-image / OpenCV / scipy.

1. Locate agent output: `/app/output/index.html`. If missing → reward = 0,
   exit cleanly.
2. **Render** with Playwright (Chromium, headless) at 1280×900,
   `full_page=True` → `agent.png`. Same browser build as was used to render
   the reference, so the comparison is apples to apples. Playwright is *not*
   doing any comparison here.
3. Resize `agent.png` and `reference.png` to a common reference height for
   fair comparison (preserve aspect within reason; record height ratio for
   the penalty term).
4. **Score** — compute the four components below using
   scikit-image / OpenCV / scipy.
5. Combine into a weighted scalar.
6. Write `reward.txt` (scalar) and `reward.json` (breakdown).

### 7.2 Components (all computed in pure Python, no Playwright)

| Component | Library / function | Weight | Notes |
|-----------|--------------------|--------|-------|
| **Visual structural similarity** | `skimage.metrics.structural_similarity` (SSIM) on grayscale, multi-scale | 0.40 | Robust to small text shifts. |
| **Colour palette match** | k-means in CIELAB → `scipy.stats.wasserstein_distance` (earth-mover's) over the cluster histograms | 0.25 | Catches palette mismatch even if layout differs. |
| **Layout IoU** | `cv2.connectedComponentsWithStats` on a grayscale-gradient mask → bounding boxes → matched IoU between sets | 0.25 | Rewards getting block placement right even with imperfect content. |
| **Aspect/height penalty** | `1 − min(1, |h_agent − h_ref| / h_ref)`, plain numpy | 0.10 | Clipped; a 50%-too-tall page gets a 0.5 multiplier on this term. |

Weights are starting points; tunable based on what correlates with human
visual judgment on the demo runs.

### 7.3 Why this combo

- **SSIM alone** over-penalises text re-flow and font hinting jitter.
- **Pixel MSE** is brittle to single-pixel shifts.
- **Pure DOM distance** is gameable (matching tag soup, wrong design).
- **VLM-as-judge** is the gold standard but adds cost, latency, and stochasticity
  inside the verifier — explicitly avoided for v1; can be a 5th component
  later as a separate axis in `reward.json`.

The 4-component ensemble is cheap (<10s on a single CPU after render),
deterministic, and gives a smooth gradient.

### 7.4 Calibration via the oracle

Running the task with `harbor run --agent oracle` should copy the reference
HTML into `/app/output/index.html`. The grader should score that ≥ 0.97.
Anything materially less means the rendering pipeline isn't deterministic
(font version drift, viewport mismatch) and we fix it before trusting scores.

## 8. Demo & visual report

### 8.1 What we run

For 1–2 final tasks (probably one cleanly diverse pair, e.g.
*landing-minimalist* and *pricing-glassmorphism*):

```
harbor run -a claude-code -m claude-opus-4-6 -k 10 -p tasks/<task>
```

10 trials per task. Cost: roughly $X-$Y on Opus 4.6 (we'll estimate before
running).

### 8.2 What the report contains

A single static HTML page (or PNG grid) showing, per task:

- The reference screenshot.
- A 2×5 grid of the 10 agent renders.
- Per-trial: weighted scalar score + the four component scores.
- Summary: mean, std, min, max, score histogram.
- Side-by-side of best vs. worst trial — the strongest visual proof of
  whether the grader's gradient matches human intuition.

### 8.3 What the report demonstrates

1. **Recipe works:** the task ran end-to-end via Harbor.
2. **Grader produces a continuous spectrum:** scores aren't all clustered at 0
   or 1; the histogram has spread.
3. **Grader correlates with visual quality:** the trial that *looks* closest
   to the reference also has the highest score.

If (3) fails on the demo, we re-tune the component weights or replace a
component before treating the recipe as complete.

## 10. Risks & mitigations

| Risk | Mitigation |
|------|------------|
| Chromium font version drift between gen and verify → grader penalises identical HTML. | Use the same Playwright base image tag everywhere. Calibrate via the oracle (§7.4). |
| LLM generates HTML that references external assets → reference render breaks at verify time (no internet in some Harbor backends). | Strip external `<img src>` URLs; allow only inline SVG and Google Fonts. Validate post-gen. |
| Agent writes HTML with `<script>`-loaded content → render at verify time looks empty. | Instruction explicitly forbids JS; grader's render also waits for `networkidle` so any allowed Google Fonts settle. |
| Grader gives near-identical scores across visually-different attempts. | Use the demo to inspect spread; tune weights or add a VLM-judge component. |
| Image too large → exceeds Claude Code's image budget. | Keep viewport ≤ 1280×900; PNG-optimize; downscale before bundling if >1 MB. |
| Anthropic API outages during generation. | Generator caches successful pages; resumable. |
